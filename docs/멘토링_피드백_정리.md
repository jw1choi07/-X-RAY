## Q1. Groundedness Check 전용 모델이 폐지된 게 맞나요?

**멘토 답변**: 공식 API 문서상으로도 잘 안 보이는 것 같다 — 별도로 확인이 어려움. 대신 **Groundedness Checking API 자체를 모델처럼 사용**해볼 것을 제안.
- 참고: https://console.upstage.ai/api/groundedness-checking

**액션**: 지금 우리가 직접 만든 "원문 대조(문자열 매칭) + Solar 재판정" 2단계 검증 로직 중, 2단계(매칭 타당성 판정)를 이 Groundedness Checking API로 교체해서 성능 비교. 자체 구현보다 공식 API가 더 안정적일 가능성.

## Q2. Structured Outputs vs 프롬프트로 JSON 강제

**멘토 답변**: 일반적으로는 프롬프트 안에 "JSON 형식으로 답하라"고 지시하는 방식을 쓰지만, **Structured Output 기능은 반드시 사용하는 게 맞다**.

**액션**: 현재 프롬프트 강제 JSON 방식 → Structured Output(JSON mode/schema)으로 전환. 파싱 실패 3건(이스케이프 오류)이 이걸로 해결될 가능성 높음 — 루브릭 "정보 처리 파이프라인 설계" 항목에도 직결.

## Q3. Document Parse가 BeautifulSoup 대비 이점이 있나요?

**멘토 답변**: 사용해볼 것을 권장. 참고 문서: https://console.upstage.ai/api/docs/for-agents/raw
**공고문처럼 표나 레이아웃이 복잡한 문서일 경우 특히 고도화 여지가 크다**고 언급.

**액션**: 지금 크롤링 문서 중 표/레이아웃이 있는 곳(약관 안 조항 표, 개인정보 위탁업체 표 등)에 Document Parse 시범 적용 → BeautifulSoup 결과와 품질 비교.

## Q4. Information Extract로 조항 단위 스키마화하면 매칭 정확도가 올라갈까요?

**멘토 답변**: AI를 활용해 문서를 **잘라서(청크) 먹이는 방식**이 좋다. 정보 수집 주기를 조절해서 크롤링을 배정하는 방식도 고려해볼 만하다.

**액션**: 통짜 텍스트를 프롬프트에 넣는 대신 조항 단위로 청크 분할 후 순차/배치 처리. 크롤링도 한 번에 몰아서 하지 말고 주기적으로 나눠 스케줄링하는 방식 고려 (서버 부하·robots.txt 우호적 측면에서도 유리).

## Q5/Q6. 매칭 타당성 통과율이 낮은 이유 / Embed 2 기반 RAG 전환

**멘토 답변** (핵심):
- 지금처럼 79개 케이스 전체를 프롬프트에 우겨넣는 방식은 **RAG 성능이 잘 안 나옴**.
- 관건은 "DB에 어떻게 쪼개서 넣을 것인가" — 정보 간 유사도를 **텍스트 기반으로만 판단하는 건 모호**하다.
- **메타데이터를 추가로 제공**해야 검색 성능이 올라간다.
- **임베딩 모델에만 의존하지 말고, 벡터DB에 데이터를 넣는 시점부터 메타데이터를 잘 설계해서 넣고, 검색 시 메타데이터 기반으로 우선 검색·필터링하는 방식을 고려**해볼 것.
- 프롬프팅을 통해 "어느 정보를 더 가져올지" 결정하게 하고, **에이전트/루프 구조로 검색된 정보의 충분성에 따라 재검색(회귀)하거나 다음 단계로 진행할지 결정**하는 에이전틱한 방식도 방법.
- **Query Rewriting / Query Expansion**으로 유저 질문(또는 우리 경우엔 "이 조항이 어떤 위험 케이스에 해당하는가" 쿼리)을 더 날카롭게 만드는 것도 고려.
  1. 질문(쿼리)의 정보 범위를 넓히고 메타데이터로 필터링
  2. 프롬프팅으로 정제

**액션**: RAG 구조 전환 시 단순 텍스트 임베딩 유사도 검색이 아니라, ToS;DR 케이스에 카테고리·위험도(weight)·서비스유형 같은 **메타데이터를 함께 인덱싱**하고, 검색 단계에서 메타데이터 필터 → 임베딩 유사도 순으로 좁혀가는 구조 설계. 필요하면 에이전틱 루프(검색 → 충분성 판단 → 재검색/진행)까지 확장.

## 벤치마크 관련 추가 조언

**Solar Pro 2 vs Solar Pro 3 비교**: Pro 3를 실제로 사용해보고 **시간 효율성(속도)을 검증해서 발표에 포함**할 것을 권장. 상황과 목적에 따라 어떤 모델이 적합한지 판단해서 근거로 제시.

**액션**: `web/scripts/benchmark-models.ts`(이미 package.json에 `npm run benchmark` 스크립트 존재)로 Pro 2 vs Pro 3 비교 실행 → 발표 자료에 속도/정확도 비교표 포함.

## 데이터 저장소 관련 조언

현재 데모는 Google Sheets를 사용 중이지만, **발표에서는 향후 DB(벡터DB 포함) 전환 계획을 설명하는 게 좋다**는 조언.

**액션**: 발표 슬라이드에 "현재: Google Sheets(프로토타입) → 향후: 벡터DB + 메타데이터 기반 검색 구조로 전환" 로드맵 한 장 추가.
