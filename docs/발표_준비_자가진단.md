## 4. 파이프라인 감사 및 개선 (2026-07-22)

"우리한테 진짜 파이프라인이라고 할 게 있나?"라는 질문에서 시작해 코드를 직접 감사함. 예상과 다른 사실들을 발견하고 실제로 고침.

**감사에서 발견한 것**
- ✅ Structured Output은 이미 구현돼 있었음 (`solar.ts`)
- ✅ Groundedness Check도 모델명 자체는 최신으로 연결돼 있는 것처럼 보였음
- ❌ 그런데 실제로 돌려보니 `solar-1-mini-groundedness-check` 모델이 **API에서 완전히 제거된 상태** — Upstage 공식 API 레퍼런스 문서(`console.upstage.ai/api/docs/for-agents/raw`)에도 Groundedness Check 엔드포인트 자체가 더 이상 없음. 멘토가 "공식 API에 잘 안 보인다"고 했던 게 정확히 맞았음.
- ❌ `rag.ts`의 "하이브리드 검색"이 실제로는 가짜 임베딩(토큰마다 값 1을 채운 배열)이었음 — `agency`/`startDate` 필드가 있는 걸로 보아 무관한 프로젝트(정부 공고문 검색)에서 그대로 복사돼 온 코드로 추정. 게다가 사용자 문서 청크에만 적용되고 정작 병목이었던 79개 ToS;DR 케이스 태그는 그대로 통째로 프롬프트에 들어가고 있었음.
- ❌ 2단계 매칭 타당성 검증(`judgeCaseMatch`, 브리핑에서 29.8% 통과율을 측정했던 그 로직)이 코드엔 있지만 **실제 서비스 어디에서도 호출되지 않고** 있었음. UI엔 뱃지 표시 로직까지 있었는데 값이 계속 비어있던 상태.

**실제로 고친 것**
1. Upstage 공식 Embeddings API(`solar-embedding-1-large-query`/`-passage`) 연동, 79개 케이스를 오프라인으로 미리 임베딩해서 캐싱
2. 문서 컨텍스트 임베딩 → 코사인 유사도로 관련 케이스 15개만 추려서 프롬프트에 전달 (79개 전량 우겨넣기 → 15개)
3. 죽어있던 매칭 타당성 검증(`judgeCaseMatch`)을 실제 분석 흐름에 연결
4. 고장난 Groundedness Check API 호출을 원래 파일럿 방법론(문자열 대조, API 호출 없음)으로 복구

**재측정 결과 (요기요·토스·카카오페이·넷플릭스·캐시슬라이드 5개, 원래 파일럿과 동일 서비스)**

| 지표 | 원래(브리핑) | 1차 수정 후(임베딩 필터링+검증 로직 복구) | 2차 수정 후(few-shot 예시 추가) |
|---|---|---|---|
| 원문근거 확인 | 64.1% | 63.3% (거의 동일 — 방법론 재현 검증됨) | **78.7%** |
| 매칭 타당성 통과 | 29.8% | 30.6% (거의 동일) | **63.8%** |

**결론**: 케이스를 79개→15개로 줄인 것만으로는 매칭 정확도가 안 올랐지만, **시스템 프롬프트에 실제 크롤링 원문에서 검증한 few-shot 예시 2개(요기요 보관기간 조항, 카카오페이 마케팅 활용 조항 — 실제 판단기준 79개 목록의 정확한 케이스 제목에 매칭시킨 예시)를 추가하자 매칭 타당성이 29.8% → 63.8%로 2배 이상 뛰었다.** 원문근거 확인율도 64.1% → 78.7%로 함께 올랐다 (few-shot이 모델의 인용 습관 자체를 더 원문 충실하게 만든 것으로 추정). 5개 서비스 47건 표본 기준이라 표본 확대 검증이 더 필요하지만, 재현성 있게 전 서비스에서 개선이 관찰됨(요기요 4→7, 토스 5→8, 카카오페이 3→6건).

## 4-1. 참고: 업스테이지 멘토링 브리핑 핵심 (원문: 약관X-ray_업스테이지_멘토링_브리핑.pdf)

- **한 줄 요약**: 아무도 안 읽는 약관을 AI가 대신 읽고 위험도 순으로 짚어주는 서비스. ToS;DR 평가 기준 참고 + 한국어 서비스 즉석 분석이 차별점.
- **문제 정의**: 이용자는 자신의 개인정보로 서비스가 무엇을 하는지 모른 채 동의하고, 기존 유일한 대안(ToS;DR)도 한국어 서비스는 커버 못함.
- **실측 근거**:
  - ToS;DR API 직접 호출 → 카카오·쿠팡·토스·무신사 등 9개 검색어 전부 DB 없음
  - 62개 문서 크롤링 성공(50개 시도), 평균 21,000자
  - 파일럿 5개 서비스 + 전체 62개 확장: 원문 근거 확인 64.1%, 매칭 타당성 통과 29.8%
  - Dropbox 사례에서 모델의 "사전지식 오염"(할루시네이션) 실제 포착
- **질문 리스트(멘토링에서 물어볼 것)**: Groundedness Check 모델 폐지 여부, Structured Output vs 프롬프트 강제 JSON, Document Parse 이점, Information Extract 조항 스키마화, 매칭 타당성 낮은 원인, Embed 2 기반 RAG 구조 전환, 크레딧 범위, rate limit, 사전지식 오염 방지 기법
