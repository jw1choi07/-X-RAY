## 자문 요청 내용
*어떤 내용을 자문받고 싶었는지 구체적으로 작성*

1. `langchain-upstage` 구버전에 명시된 Groundedness Check 전용 모델(`solar-1-mini-answer-verification`)이 현재 API에서 제거되어 `invalid_request_error`가 발생함을 확인함. 공식적으로 폐지된 것이 맞는지, 대체 권장 방식이 있는지.
2. "원문 인용 + 판단 근거"를 강제하는 구조화 출력을 안정적으로 뽑으려면 Structured Outputs/JSON mode를 쓰는 게 나은지, 지금처럼 프롬프트로 JSON을 강제하는 방식이 괜찮은지.
3. Document Parse를 아직 실전 투입하지 않았는데, 현재 BeautifulSoup으로 뽑은 텍스트 대비 실제 이점이 있는지 (표/레이아웃이 복잡한 문서에서 특히 유용한지).
4. Information Extract로 "조항 단위"를 스키마 기반으로 미리 잘라낸 뒤 Solar에 넘기면, 통짜 텍스트를 넘기는 것보다 매칭 정확도가 올라갈지.
5. 매칭 타당성 통과율이 서비스별로 10~40%대로 낮은 편인데, 프롬프트 구조 문제인지 모델 한계인지.
6. Embed 2(임베딩 API)로 조항 텍스트와 ToS;DR case 설명을 벡터 유사도로 1차 필터링한 뒤 Solar에 넘기는 RAG형 구조가, 지금의 "79개 케이스 전체를 프롬프트에 우겨넣는" 방식보다 나을지.
7. Dropbox 사례에서 모델이 원문에 없는 내용(사전 지식)을 지어내는 현상을 재현함 — "제공된 텍스트만 근거로 삼으라"는 프롬프트 지시 외에, 이런 "사전 지식 오염" 방지를 위해 권장하는 기법이 있는지.
8. Solar Pro 2로 파일럿을 돌렸는데, Pro 3의 향상된 reasoning이 매칭 타당성 통과율을 실질적으로 올려줄지.

## 자문 내용
*어떤 내용을 자문받았는지 구체적으로 작성*

**1. Groundedness Check 관련**
공식 API 문서상으로도 잘 보이지 않는 것으로 확인됨(멘토도 명확한 폐지 공지는 찾지 못함). 별도의 전용 모델을 계속 찾기보다, **Groundedness Checking API**(`console.upstage.ai/api/groundedness-checking`)를 자체 검증 모델처럼 직접 사용해볼 것을 제안받음.

**2. Structured Output**
일반적으로는 프롬프트에 "JSON 형식으로 답하라"고 지시하는 방식을 쓰기도 하지만, **Structured Output 기능은 반드시 사용하는 것이 맞다**는 답변. 현재 겪고 있는 JSON 파싱 실패(이스케이프 오류) 문제도 이 방식 전환으로 해결될 가능성이 높다고 언급.

**3. Document Parse**
사용해볼 것을 권장받음(`console.upstage.ai/api/docs/for-agents/raw`). 특히 **표나 레이아웃이 복잡한 문서(예: 공고문 형태)일 경우 Document Parse로 고도화하는 것이 효과적**이라는 조언.

**4. Information Extract / 청크 전략**
AI를 활용해 문서를 **잘라서(청크 단위로) 순차 처리**하는 방식이 좋다는 조언. 크롤링 자체도 한 번에 몰아서 하기보다, 수집 주기를 조절해서 배정하는 방식을 고려해볼 만하다는 언급.

**5-6. RAG 성능 및 매칭 타당성 저조 원인**
- 지금처럼 79개 케이스 전체를 프롬프트에 우겨넣는 방식은 RAG 성능이 잘 나오지 않는 구조라는 진단.
- 핵심 쟁점은 "벡터DB에 정보를 어떻게 쪼개서 넣을 것인가" — 정보 간 유사도를 **텍스트 기반으로만 판단하는 것은 모호**하며, **메타데이터를 추가로 설계해서 함께 제공**해야 검색 성능이 올라간다는 조언.
- **임베딩 모델 자체에만 의존하지 말고, 벡터DB에 데이터를 적재하는 시점부터 메타데이터(카테고리, 위험도 등)를 잘 설계해서 넣고, 검색 시 메타데이터 기반으로 우선 검색·필터링한 뒤 임베딩 유사도로 좁혀가는 구조**를 권장.
- 프롬프팅을 통해 "어떤 정보를 더 가져올지"를 스스로 판단하게 하고, **에이전트/루프 구조로 검색된 정보의 충분성에 따라 재검색하거나 다음 단계로 진행할지 결정하는 에이전틱한 방식**도 하나의 방법으로 제시됨.
- **Query Rewriting / Query Expansion**으로 검색 쿼리를 더 정교하게 다듬는 방법도 언급 — (1) 질문의 정보 범위를 넓히고 메타데이터로 필터링하는 방향, (2) 프롬프팅으로 정제하는 방향 두 가지로 접근 가능.

**7. 사전 지식 오염 방지**
위 Groundedness Checking API 활용 및 원문 대조 검증 구조 자체가 답변에 해당 — 별도의 특별한 기법보다는 검증 단계를 확실히 두는 것이 핵심이라는 뉘앙스.

**8. Solar Pro 2 vs Pro 3**
직접 Pro 3를 사용해보고 **시간 효율성(속도)을 검증해서 발표 자료에 근거로 포함**할 것을 권장받음. 상황과 목적에 따라 어떤 모델이 적합한지는 실측 비교를 통해 판단하는 것이 바람직하다는 조언.

**추가 조언 (데이터 저장소)**
현재 데모는 Google Sheets를 활용 중이지만, 발표에서는 **향후 벡터DB 등으로 전환할 계획을 명확히 설명하는 것이 좋다**는 조언을 받음.
